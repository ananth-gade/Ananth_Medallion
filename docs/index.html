🏗️ RS Medallion — Developer Playbook

A generic, reusable guide for building data pipelines with the Medallion Architecture in dbt + Snowflake.
Every pattern shown here is illustrated with real reference implementations from the repository.

📖 How to Use This Guide

Each section describes the generic pattern first, followed by a ◼ Reference Implementation block that shows the actual model from this repo. When building a new entity, copy the template, swap in your entity name, and follow the same structure.

📋 Overview

This playbook documents the patterns, macros, and conventions used to build end-to-end data pipelines in the RS Medallion repository. The framework follows a four-layer Medallion Architecture:

Database Routing

Each layer routes to a separate Snowflake database using the generate_schema_name macro in dbt_project.yml:

# dbt_project.yml — database routing pattern
models:
  rs_medallion:
    1_bronze:
      {source_name}:              # e.g. workday, es_eqai
        +database: "{SOURCE_DB}"  # WORKDAY, EQAI
        +schema: ODS
    2_silver:
      cdm:
        {domain}:                 # e.g. workforce
          +database: CDM
          +schema: "{DOMAIN}"     # WORKFORCE
    3_gold:
      core:
        {sub_domain}:             # e.g. shared_dims, workforce
          +database: CORE
          +schema: "{SUB_DOMAIN}"

Bronze ODS Layer — Template

The Bronze layer ingests raw data from source systems with minimal transformation. Each source table maps 1:1 to an ODS model. The goal is to create an auditable, hash-tracked, incrementally loaded copy of the source.

Standard Configuration Block

Every Bronze model must include these config keys:

Template 
{{
    config(
        materialized   = 'incremental',
        unique_key     = 'ODS_{SOURCE}_{ENTITY}_PK',
        tags           = ["bronze", "{source}", "{domain}"],
        on_schema_change = 'append_new_columns',

        -- Hash-based merge: only update when data changes
        merge_condition = 'DBT_INTERNAL_DEST.HASH_FULL <> DBT_INTERNAL_SOURCE.HASH_FULL',

        -- Pre-hook: deduplicate staging before merge
        pre_hook  = "{{ purge_staging_duplicates_by_hash(
                        '{SOURCE_DB}.STG.STG_{SOURCE}_{ENTITY}',
                        ['NATURAL_KEY_COL1','NATURAL_KEY_COL2'],
                        'HASH_FULL') }}",

        -- Post-hook: propagate source deletes as soft-deletes
        post_hook = "{{ mark_deleted_records(
                        this,
                        '{SOURCE_DB}.STG.STG_{SOURCE}_{ENTITY}',
                        ['NATURAL_KEY_COL1','NATURAL_KEY_COL2'],
                        'DQ_ACTIVE_FLG') }}"
    )
}}

Standard CTE Pattern

Bronze models follow a 3-CTE structure: source → transform → final.

Template 
-- CTE 1: Source — pull from staging
WITH source AS (
    SELECT * FROM {{ source('{source_name}', 'STG_{SOURCE}_{ENTITY}') }}
    {% if is_incremental() %}
    WHERE METADATA_TIMESTAMP >= {{ get_max_event_time('METADATA_TIMESTAMP') }}
    {% endif %}
),

-- CTE 2: Transform — add keys, hashes, audit columns
transformed AS (
    SELECT
        -- Surrogate primary key
        {{ generate_surrogate_key(['NATURAL_KEY_COL1', 'NATURAL_KEY_COL2']) }}
            AS ODS_{SOURCE}_{ENTITY}_PK,

        -- Source columns (trim strings)
        {{ trim_string('COL_A') }}          AS COL_A,
        {{ trim_string('COL_B') }}          AS COL_B,
        CAST(COL_C AS DATE)                 AS COL_C,

        -- Hash columns for CDC
        {{ generate_hash_change(['TYPE2_COL_A', 'TYPE2_COL_B']) }} AS HASH_CHANGE,
        {{ generate_hash_full(['COL_A', 'COL_B', 'COL_C']) }}     AS HASH_FULL,

        -- Audit metadata
        '{{ invocation_id }}'               AS ODS_INS_BATCH_ID,
        '{{ invocation_id }}'               AS ODS_UPD_BATCH_ID,
        CURRENT_TIMESTAMP()                 AS ODS_INSERT_TIMESTAMP,
        CURRENT_TIMESTAMP()                 AS ODS_UPDATE_TIMESTAMP,
        'I'                                 AS DQ_OPERATION_TYPE,
        'Y'                                 AS DQ_ACTIVE_FLG,
        CURRENT_TIMESTAMP()                 AS METADATA_TIMESTAMP

    FROM source
)

SELECT * FROM transformed

Silver CDM Layer — Template

The Silver layer creates a Conformed Data Model (CDM) — a single, clean, business-named entity that may combine multiple Bronze sources. This is where cross-source enrichment, type standardization, and business logic consolidation happen.

Standard Configuration Block

Template 
{{
    config(
        materialized   = 'incremental',
        unique_key     = '{ENTITY}_PK',
        tags           = ["silver", "{domain}"],
        on_schema_change = 'append_new_columns',

        merge_condition = 'DBT_INTERNAL_DEST.HASH_FULL <> DBT_INTERNAL_SOURCE.HASH_FULL'
    )
}}

Standard CTE Pattern

Silver models use a multi-CTE pattern: primary source → secondary sources → enrichment joins → transformations → final.

Template 
-- CTE 1: Primary Bronze source
WITH primary_source AS (
    SELECT * FROM {{ ref('ODS_{SOURCE}_{ENTITY}') }}
    WHERE DQ_ACTIVE_FLG = 'Y'  -- exclude soft-deletes
    {% if is_incremental() %}
    AND ODS_UPDATE_TIMESTAMP >= {{ get_max_event_time('CDM_UPDATE_TIMESTAMP') }}
    {% endif %}
),

-- CTE 2+: Secondary Bronze sources (for enrichment)
secondary_source AS (
    SELECT * FROM {{ ref('ODS_{SOURCE2}_{ENTITY2}') }}
    WHERE DQ_ACTIVE_FLG = 'Y'
),

-- CTE: Enriched — join primary with secondary
enriched AS (
    SELECT
        p.*,
        s.ENRICHMENT_COL_A,
        s.ENRICHMENT_COL_B
    FROM primary_source p
    LEFT JOIN secondary_source s
        ON p.JOIN_KEY = s.JOIN_KEY
),

-- CTE: Transformed — apply business naming, type casting, derivations
transformed AS (
    SELECT
        -- Surrogate PK for this layer
        {{ generate_surrogate_key(['NATURAL_KEY']) }} AS {ENTITY}_PK,

        -- Business-named columns
        CAST(SRC_COL_A AS VARCHAR)              AS BUSINESS_NAME_A,
        CAST(SRC_COL_B AS DATE)                 AS BUSINESS_NAME_B,
        COALESCE(ENRICHMENT_COL_A, 'Unknown')   AS ENRICHED_FIELD,

        -- Hash columns for downstream
        {{ generate_hash_change(['BUSINESS_NAME_A', 'BUSINESS_NAME_B']) }} AS HASH_CHANGE,
        {{ generate_hash_full(['BUSINESS_NAME_A', 'BUSINESS_NAME_B', 'ENRICHED_FIELD']) }} AS HASH_FULL,

        -- CDM audit columns
        '{{ invocation_id }}'       AS CDM_INS_BATCH_ID,
        '{{ invocation_id }}'       AS CDM_UPD_BATCH_ID,
        CURRENT_TIMESTAMP()         AS CDM_INSERT_TIMESTAMP,
        CURRENT_TIMESTAMP()         AS CDM_UPDATE_TIMESTAMP

    FROM enriched
)

SELECT * FROM transformed

Gold Dimension (SCD Type 2) — Template

Gold Dimensions implement Slowly Changing Dimension Type 2 using the custom scd_incremental materialization. Each change to a tracked attribute creates a new version row with effective/discontinue dates.

Standard Configuration Block

Template 
{{
    config(
        materialized         = 'scd_incremental',
        unique_key           = '{ENTITY}_SK',
        incremental_strategy = 'scd+merge',
        tags                 = ["gold", "{domain}", "core-dims"],
        on_schema_change     = 'append_new_columns',

        -- SCD2 engine config
        timestamp_column     = 'CORE_UPDATE_TIMESTAMP',
        dest_columns         = ['HASH_CHANGE', 'DBT_HASH', 'DISC_DT', 'IS_CURRENT'],
        merge_condition      = 'DBT_INTERNAL_DEST.DBT_HASH <> DBT_INTERNAL_SOURCE.DBT_HASH',
        matched_condition    = 'DBT_INTERNAL_DEST.IS_CURRENT = TRUE',

        -- Post-hook: mark discontinued rows from soft-deleted source records
        post_hook = "UPDATE {{ this }}
                     SET IS_CURRENT = FALSE,
                         DISC_DT    = CURRENT_DATE
                     WHERE {NATURAL_KEY} NOT IN (
                         SELECT {NATURAL_KEY}
                         FROM {{ ref('{ENTITY}') }}
                     )
                     AND IS_CURRENT = TRUE"
    )
}}

Standard CTE Pattern

Template 
WITH source AS (
    SELECT * FROM {{ ref('{ENTITY}') }}  -- Silver CDM model
    {% if is_incremental() %}
    WHERE CDM_UPDATE_TIMESTAMP >= {{ get_max_event_time('CORE_UPDATE_TIMESTAMP') }}
    {% endif %}
),

transformed AS (
    SELECT
        -- Surrogate key: natural key + effective date → unique per version
        {{ generate_surrogate_key(['NATURAL_KEY', 'EFF_DT']) }} AS {ENTITY}_SK,

        -- Natural / business key (carried from CDM)
        NATURAL_KEY,

        -- Dimension attributes
        ATTRIBUTE_A,
        ATTRIBUTE_B,
        ATTRIBUTE_C,

        -- Hash columns
        HASH_CHANGE,
        HASH_CHANGE AS DBT_HASH,  -- mapped for SCD engine

        -- SCD2 metadata
        CURRENT_DATE                AS EFF_DT,
        CAST('9999-12-31' AS DATE)  AS DISC_DT,
        TRUE                        AS IS_CURRENT,

        -- Gold audit columns
        '{{ invocation_id }}'       AS CORE_INS_BATCH_ID,
        '{{ invocation_id }}'       AS CORE_UPD_BATCH_ID,
        CURRENT_TIMESTAMP()         AS CORE_INSERT_TIMESTAMP,
        CURRENT_TIMESTAMP()         AS CORE_UPDATE_TIMESTAMP

    FROM source
)

SELECT * FROM transformed

How the SCD2 Engine Works

When the scd+merge strategy executes:

  1. MATCH + CHANGED: The existing current row's IS_CURRENT → FALSE and DISC_DT → today (via merge.sql)
  2. INSERT: New version row inserted with IS_CURRENT = TRUE, EFF_DT = today, DISC_DT = 9999-12-31
  3. MATCH + UNCHANGED: Only audit timestamps are updated (no new version)
  4. NOT MATCHED: Brand-new record inserted as first version
Reference Implementation

DIM_EMPLOYEE · models/3_gold/core/shared_dims/DIM_EMPLOYEE.sql

  • Source: {{ ref('EMPLOYEE') }} (Silver CDM)
  • Surrogate key: EMPLOYEE_SK = MD5(EMPLOYEE_ID || EFF_DT)
  • Tracked attributes: Name, Email, Job Code, Manager, Status, Location, Compensation, etc.
  • Post-hook: Discontinues employees deleted from source
  • SCD2 columns: EFF_DT, DISC_DT, IS_CURRENT

Gold Fact Table — Template

Fact tables derive measurable events from SCD2 dimension versions. They use LAG() window functions to compare consecutive versions and detect what changed.

Standard Configuration Block

Template 
{{
    config(
        materialized = 'table',          -- full refresh each run
        tags         = ["gold", "{domain}", "core-facts"]
    )
}}

Event Detection Pattern

Template 
WITH dim_versions AS (
    SELECT
        *,
        -- Compare to previous version
        LAG(ATTRIBUTE_A) OVER (PARTITION BY NATURAL_KEY ORDER BY EFF_DT)  AS PREV_ATTRIBUTE_A,
        LAG(ATTRIBUTE_B) OVER (PARTITION BY NATURAL_KEY ORDER BY EFF_DT)  AS PREV_ATTRIBUTE_B,
        LAG(EFF_DT)      OVER (PARTITION BY NATURAL_KEY ORDER BY EFF_DT)  AS PREV_EFF_DT,
        ROW_NUMBER()      OVER (PARTITION BY NATURAL_KEY ORDER BY EFF_DT)  AS VERSION_NUM
    FROM {{ ref('DIM_{ENTITY}') }}
),

events AS (
    SELECT
        {{ generate_surrogate_key(['NATURAL_KEY', 'EFF_DT', 'EVENT_TYPE']) }}
            AS FACT_{ENTITY}_{PROCESS}_PK,
        NATURAL_KEY,

        -- Derived event type
        CASE
            WHEN VERSION_NUM = 1                           THEN 'INITIAL'
            WHEN ATTRIBUTE_A <> PREV_ATTRIBUTE_A           THEN 'ATTRIBUTE_A_CHANGE'
            WHEN ATTRIBUTE_B <> PREV_ATTRIBUTE_B           THEN 'ATTRIBUTE_B_CHANGE'
            ELSE 'OTHER_CHANGE'
        END AS EVENT_TYPE,

        -- Measures
        DATEDIFF('day', PREV_EFF_DT, EFF_DT)              AS DAYS_SINCE_LAST_CHANGE,

        -- Event date
        EFF_DT                                              AS EVENT_DATE,

        -- Audit
        '{{ invocation_id }}'                               AS CORE_INS_BATCH_ID,
        CURRENT_TIMESTAMP()                                 AS CORE_INSERT_TIMESTAMP

    FROM dim_versions
)

SELECT * FROM events

Audit Layer

The Audit layer captures metadata about every dbt run — row counts, execution times, warnings, errors — for observability and debugging.

on-run-start: The audit() macro creates AUDIT.DBT_MONITORING.DBT_RUN_RESULTS if it doesn't exist
Model execution: dbt runs all models and tests as normal
on-run-end: The capture_run_results() macro iterates over results and inserts one row per node into the audit table

Every entity follows the same data flow pattern through the four layers:

Source System → Staging
External process (ELT tool / ADF / Fivetran) lands raw data in {SOURCE_DB}.STG.STG_{SOURCE}_{ENTITY}

📚 SCD Type 1 vs Type 2

Understanding when to use each SCD type is fundamental to the medallion architecture.

AspectSCD Type 1 (Overwrite)SCD Type 2 (Version)
StrategyMerge — update in placescd+merge — insert new version
HistoryNo history keptFull history with EFF_DT / DISC_DT
Use caseBronze & Silver layers (latest state)Gold dimensions (audit trail)
Materializationincrementalscd_incremental
Change detectionHASH_FULL in merge_conditionDBT_HASH (mapped from HASH_CHANGE)
Example layerODS_WORKDAY_EMPLOYEE, EMPLOYEEDIM_EMPLOYEE
Two Hash Columns, Two Purposes

HASH_CHANGE — Covers only the columns you want to track for SCD Type 2 versioning (e.g., job code, status, compensation). If this changes, a new dimension version is created.

HASH_FULL — Covers all business columns. Used as the merge condition in Bronze/Silver to avoid unnecessary updates when no data has changed.

incremental (merge)Bronze, SilverStandard dbt merge — inserts new rows, updates changed rows based on unique_key + merge_condition

📈 Incremental Strategy

All incremental models use a watermark pattern with the get_max_event_time() macro:

{% if is_incremental() %}
WHERE METADATA_TIMESTAMP >= {{ get_max_event_time('METADATA_TIMESTAMP') }}
{% endif %}

This pulls only rows newer than the latest timestamp already in the target table, ensuring efficient incremental loads.

Merge Condition

The merge_condition config prevents unnecessary updates:

merge_condition = 'DBT_INTERNAL_DEST.HASH_FULL <> DBT_INTERNAL_SOURCE.HASH_FULL'

Even if a row is re-selected by the watermark filter, it won't be updated unless the data has actually changed.

Full Refresh

Use --full-refresh to drop and recreate incremental tables from scratch. This is needed after schema changes or when backfilling historical data.

#️⃣ Hash-Based Change Data Capture

Every layer uses MD5 hashes for change detection. This pattern is central to the framework:

🔧 SCD Merge Engine — Deep Dive

The custom scd_incremental materialization (macros/scd_engines/) implements the SCD2 merge in two files:

snowflake_merge.sql (Materialization)

Wraps the standard dbt incremental materialization with SCD-specific hooks. It calls merge.sql to generate the actual MERGE statement.

merge.sql (MERGE Statement Builder)

Generates a Snowflake MERGE with these clauses:

  1. WHEN MATCHED + changed: Update IS_CURRENT = FALSE, DISC_DT = CURRENT_DATE on the existing row
  2. WHEN MATCHED + unchanged: Update only audit timestamps (no new version)
  3. WHEN NOT MATCHED: Insert new row with IS_CURRENT = TRUE, DISC_DT = '9999-12-31'
  4. Post-merge INSERT: For changed rows, insert a new version with fresh EFF_DT and IS_CURRENT = TRUE

Config Parameters

ParameterPurposeTypical Value
incremental_strategyActivates SCD engine'scd+merge'
timestamp_columnAudit timestamp for version ordering'CORE_UPDATE_TIMESTAMP'
dest_columnsColumns managed by SCD engine (not from source)['HASH_CHANGE','DBT_HASH','DISC_DT','IS_CURRENT']
merge_conditionDetermines if data changed'DBT_INTERNAL_DEST.DBT_HASH <> ...'
matched_conditionTargets only current version for update'DBT_INTERNAL_DEST.IS_CURRENT = TRUE'

🔧 Helper Macros — Usage Guide

generate_surrogate_key

Creates an MD5 hash from one or more columns. Use for all primary and surrogate keys:

-- Single column key
{{ generate_surrogate_key(['EMPLOYEE_ID']) }} AS ODS_WORKDAY_EMPLOYEE_PK

-- Composite key (dimension: natural key + effective date)
{{ generate_surrogate_key(['EMPLOYEE_ID', 'EFF_DT']) }} AS EMPLOYEE_SK

generate_hash_change / generate_hash_full

MD5 hashes for change detection. Pass a list of column names:

-- Hash of SCD2-tracked columns only
{{ generate_hash_change(['JOB_CD', 'STATUS', 'MANAGER_ID', 'COMPENSATION']) }} AS HASH_CHANGE

-- Hash of ALL business columns
{{ generate_hash_full(['COL_A', 'COL_B', 'COL_C', '...']) }} AS HASH_FULL

get_max_event_time

Returns the MAX value of a timestamp column from the target table (for incremental watermark):

{% if is_incremental() %}
WHERE METADATA_TIMESTAMP >= {{ get_max_event_time('METADATA_TIMESTAMP') }}
{% endif %}

trim_string

Trims whitespace and converts empty strings to NULL:

{{ trim_string('FIRST_NAME') }} AS FIRST_NAME

Called in on-run-start. Creates the audit monitoring table if it doesn't exist. No parameters needed — it uses get_audit_location() internally.

🎯 Singular Tests (Per-Model)

Located in tests/singular/{model_name}/. These are standalone SQL files that return failing rows. Create them for cross-layer reconciliation and domain-specific business rules.

Cross-Layer Reconciliation Pattern

For every layer transition, create these three reconciliation tests:

Template 
-- tests/singular/{model}/test_{model}_row_count.sql
-- Validates row count consistency between layers
SELECT
    (SELECT COUNT(*) FROM {{ ref('SOURCE_MODEL') }}) AS source_count,
    (SELECT COUNT(*) FROM {{ ref('TARGET_MODEL') }}) AS target_count
WHERE source_count <> target_count
Template 
-- tests/singular/{model}/test_{model}_pk_reconciliation.sql
-- Validates all PKs from source exist in target
SELECT s.NATURAL_KEY
FROM {{ ref('SOURCE_MODEL') }} s
LEFT JOIN {{ ref('TARGET_MODEL') }} t
    ON s.NATURAL_KEY = t.NATURAL_KEY
WHERE t.NATURAL_KEY IS NULL
Template 
-- tests/singular/{model}/test_{model}_referential_integrity.sql
-- Validates FK relationships between fact and dimension
SELECT f.DIMENSION_KEY
FROM {{ ref('FACT_TABLE') }} f
LEFT JOIN {{ ref('DIM_TABLE') }} d
    ON f.DIMENSION_KEY = d.NATURAL_KEY
WHERE d.NATURAL_KEY IS NULL

🛠️ Developer Workflow — Adding a New Entity

Follow this step-by-step checklist when adding any new entity to the medallion architecture. Each step builds on the previous one, and you should test at each layer before moving to the next.

Define the Source
Create models/1_bronze/{source}/z_{source}_sources.yml defining the source database, schema, and table(s). If the source system folder doesn't exist yet, create it.
Build the Bronze (ODS) Model
Create models/1_bronze/{source}/ODS_{SOURCE}_{ENTITY}.sql following the Bronze Template. Include: surrogate key, hash columns, audit columns, pre-hook (dedup), post-hook (soft-delete). Create the schema YAML with column docs and generic tests.
Test: dbt build --select ODS_{SOURCE}_{ENTITY}
Build the Silver (CDM) Model
Create models/2_silver/cdm/{domain}/{ENTITY}.sql following the Silver Template. Join multiple Bronze sources if needed, apply business naming, and generate fresh hash columns. Create schema YAML.
Test: dbt build --select {ENTITY}
Build the Gold Dimension (SCD Type 2)
Create models/3_gold/core/{sub_domain}/DIM_{ENTITY}.sql following the Gold Dimension Template. Use scd_incremental materialization, map HASH_CHANGE → DBT_HASH, add SCD2 metadata columns. Create schema YAML.
Test: dbt build --select DIM_{ENTITY}
Build the Gold Fact Table (Optional)
Create models/3_gold/core/{sub_domain}/FACT_{ENTITY}_{PROCESS}.sql following the Gold Fact Template. Use LAG() window functions to detect events from dimension versions. Create schema YAML.
Test: dbt build --select FACT_{ENTITY}_{PROCESS}
Add Cross-Layer Singular Tests
Create test files in tests/singular/{model}/ for row count reconciliation, PK reconciliation, and referential integrity. See Singular Tests.
Update dbt_project.yml
Add database/schema routing for the new source/domain under the appropriate layer. See Architecture → Database Routing.
Run the Full Lineage
dbt build --select +FACT_{ENTITY}_{PROCESS}
The + prefix runs all upstream dependencies. Verify all models pass and all tests are green.
✅ Definition of Done
  • All 4 layers built and passing (Bronze → Silver → Gold Dim → Gold Fact)
  • Schema YAML with column documentation and generic tests for each model
  • Singular tests for cross-layer reconciliation at every layer transition
  • SCD2 integrity tests on dimension (no overlapping dates, one current row per key)
  • dbt build --select +FACT_{ENTITY}_{PROCESS} completes with 0 errors
  • Database routing configured in dbt_project.yml
